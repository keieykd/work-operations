📝 LINE Pay 結算與撥款業務手冊

1. 會議摘要

核心聚焦於 LINE Pay 第三方支付與電子支付的金流底層邏輯。從消費者「授權(Auth)」、商家「請款(Capture)」，到系統生成「撥款底稿(Settle)」並打包「結算(Aggregation)」，最終產出傳票完成撥款。重點釐清了日結、週結、月結遇上「假日」的系統排程差異（日結順延，週/月結假日照跑），以及針對不同商戶（如 7-11 統整請款、Uber 僅做資訊分潤、新光三越的 MID 邏輯）的特殊處理規則，為後續複雜的財務對帳打下堅實的觀念基礎。

【第一部分：核心觀念與架構】

2. 完整工作流程 (Transaction Flow)

  1. 授權 (Auth)：消費者刷卡，收單行發出通知。此階段僅「圈存」額度，尚未扣款。
  2. 請款 (Capture)：交易確認成立，商家向系統請款。資料正式進入 LINE Pay 系統。
  3. 撥款底稿 (Settle)：Capture 發生後約 1 秒內，系統瞬間逐筆生成應撥款的「原始資料」。
  4. 結算 (Aggregation)：系統依據商家合約週期（日/週/月），將多筆 Settle 打包成「一整包」。
  5. 傳票 (Journal / JE)：結算完成後，系統產出對應的財務撥款傳票金額。
  6. 撥款 (Settlement)：財務端核對傳票與實際銀行金流無誤後，將資金撥付至商家帳戶。(註：Payout 通常是指商店跟用戶之間的金流英文，撥款給商家通用語為 Settlement)。

3. 金句與雷區 (Tips & Red Zones)

💡 金句：「我們部門要做的核心，就是確保 Capture 資料都有到 Settle，Settle 有結算出正確的傳票 (JE)，最後傳票金額跟實際銀行出去的金流要一模一樣。」

🚫 絕對不能犯的禁忌與雷區：
  1. 搞錯假日的「結算」邏輯：
    日結遇假日會順延；但是 「週結」跟「月結」不管是不是假日，系統都會硬生生結算下去！
    👉 長假防禦機制：過年或長假前，務必先把系統會在假日自動結算的電子傳票，預先送交財務系統排程，否則連假回來第一天會撥款大當機。
  2. 忘記「撥款日必定是工作日」：
    如果商戶是 W111（週一結算，週二撥款），但週二是國定假日，這筆錢絕對不會在週二出去，系統會自動順延到週三（下一個工作日）撥款。
  3. 搞混「代收代付」的法規限制：
    第三方支付原則上不能做代收代付業者與代收代付業者的介接。但電子支付法規有開後門，允許「外送平台」與「計程車」這兩種特例進行代收代付介接。
    因此，若不能幫純外送平台（如 Uber、FoodPanda）做全額代收代付，對 Uber 我們僅做「資訊串接與分潤抽成」，不碰本金；但如果是 Momo（買貨進來再賣出去的實質買賣），我們就可以完整代收代付。

4. 我的待辦事項

  • 高優先 熟記並實際操作「每日結算統計報表 SOP (12 步驟)」。
  • 中等 閱讀內部 LP 維基百科，查詢各類結算週期 (W110, A1 等) 的邏輯。

5. 專有名詞小字典

專有名詞白話解釋
第三/電子支付差異 第三方支付 (TSP)：只有信用卡支付。需串接收單行(玉山/台新等五大行)，錢從銀行來。
電子支付：區分成用信用卡支付（有收單）或用儲值金支付（無收單，直接扣款）。
★ 系統判定邏輯：消費時，若商家與用戶雙方皆有申請電子支付，系統即走電子支付；若有一端未申請，就會走 TSP (第三方支付)。
Auto / Manual CaptureAuto：授權後立刻請款。Manual：非即時請款。例如：7-11 交易量太大一天包成一包請款；NET 每 4 小時請款一次；Uber 等餐點送達後才請款。
MID (Merchant ID)商店代號。同集團（如新光三越）統編相同，但為區分不同電腦或櫃位，會擁有多個不同 MID，通常前綴號碼會一樣。
日結 (A1, A2...)A1 = 次 1 日撥款；A2 = 次 2 日撥款。★ 日結僅在工作日進行結算。
週結 (W110, W111...)W 代表週結，統一結算上週一到週日的帳。W110 = 週一當天撥款；W111 = 週二撥款。
月結1 號到月底的帳，統一在次月 1 號結算（次月 2 號或 5 號撥款）。

6. 各步驟預估時間與難度

  • 理解底層金流架構 (Auth/Capture/Settle)：★ 簡單（理解邏輯即可）
  • 辨識特殊商戶請款與分潤邏輯 (Uber / Momo / 7-11)：★★ 中等（需記憶特例）
  • 掌握結算週期與假日順延機制 (A1 / W110)：★★★ 需要練習（需拿著月曆推演連假狀況）
  • 執行每日結算報表與抓帳 (下方 12 步驟)：★★★ 需要練習（Excel 步驟極多，需高度專注力）

🚀 7. 一頁版快速複習卡 (濃縮精華)

🔹 核心對帳公式：
Capture (請款) ＝ Settle (撥款底稿) ＝ Aggregation (結算) ＝ Journal/JE (傳票) ＝ 銀行實際撥款 (Settlement)

🔹 假日生存法則：
【結算日】：日結遇紅字順延；週/月結遇紅字照樣結算。
【撥款日】：所有撥款永遠只在工作日發生，遇紅字一律往後順延！

🔹 誰給錢最快？ 台新銀行 T+1 款；其他收單行 T+2 款。

【第二部分：每日結算統計報表 SOP (實戰操作)】

前情提要：假設今天為 02/27，要做今天的結算統計報表。操作分為「下載三份原始檔」與「整合至當天撥款報表模板」兩大階段。

一、 下載三份來源檔案 (Raw Data)

將以下三份從 LINE Pay 內部系統下載的 Excel 檔案置於同一個路徑中：

1. 下載傳票 (Search for Merchant Settlement)
  • 路徑：Admin Function > Settlement Confirm > Search for Merchant Settlement
  • 搜尋條件：
    • Aggregation date: 選擇當天日期 02/27
    • Scheduled Settlement date: 選擇範圍為 當天 02/27 - 往後兩個月 (例如 04/30) (主要是要把最久的 A25 情況包含進來)
  • 操作：按下 Search 生成結果後，點選 Download 下載結算傳票 Excel。
2. 商家統計數據 (Merchant Statistics)
  • 用途：抓取每個商家的統計數據，比對請款與撥款是否一致。
  • 路徑：Settlement > Settlement Statistics > Merchant Statistics
  • 搜尋條件 (交易日)：搜尋前一天 (例如 02/26)。
    ★ 特殊規則：如果前一天是假日，則要包含前一個最後一個工作日，以及中間包含的所有假日 (例：今天是週一，則選擇上週五~週日的日期範圍)。
  • 操作：點擊黃色的 Bulk download 鈕。在跳出的 dialog 選擇日期範圍，必須勾選 Delete previous Aggregation file。點選下載後，算是非同步下載，須至 Admin 下的 Bulk Data download 清單等待寫入完成並顯示下載按鈕。
3. 逐筆撥款明細統計 (Statistics for validation)
  • 路徑：Settlement > Settlement Statistics > Statistics for validation
  • 搜尋條件 (日期範圍)：選擇結算日當天 02/27。
  • 操作：同樣點擊黃色的 Bulk download 鈕，選擇日期範圍並勾選 Delete previous Aggregation file。至 Bulk Data download 清單等待下載完成。

二、 整合至「當天的撥款報表」Excel

以下操作皆在既有的模板「當天的撥款報表」中執行（已內建完整分頁名稱、表頭與樞紐分析表 Pivot）。

Step 1. 處理「逐筆撥款明細」 (排除後結手續費)

  1. 打開【檔案 3：逐筆撥款明細】檔案，把 Sheet1 全部的值貼到模板的分頁 Settlement Statistics 中（連表頭都一樣不變）。
  2. 將 Column Fee Charge Method 篩選出值為 Payable。
  3. 接著把 Service Fee 跟 VAT 這兩個 Column 篩選出來的每個 rows 的金額都刪除（因為這算是手續費後結，不放進現在的資料表內）。

Step 2. 處理「結算傳票」(篩選 AP 應付與整理格式)

  1. 打開【檔案 1：結算傳票】Excel 進行資料篩選：
    • 篩選表頭 NEON Fees Field 欄位：篩選出包含 Fee 字串（代表 AP-應付），不勾選空白的選項。
    • 篩選表頭 Transferred(Y/N) 欄位：選擇 Y（代表已傳送至 NEON 系統，應付為正數）。
    • 篩選表頭 I/F Code 欄位：僅挑出 LPTW_EPI_Payment 及 LPTW_EPI_Payment_Aft，只選這兩個，其他不選。
  2. 將這個表格的全部篩選結果，貼到模板的分頁 Finance IF (Admin JE)。
  3. 貼上後，再次在該分頁篩選表頭 I/F code 為 LPTW_EPI_Payment_Aft，接著把 Service Fee 跟 VAT 這兩個 Column 的每個 rows 金額都刪除（同樣為手續費後結）。
  4. 將 Column 表頭 Actual Deposit Amount 的每個 rows 的值改為數字格式（貼過來時會是文字格式，每個欄位左上角會有綠色三角形，需轉換）。

Step 3. 處理「退款準備金 (Refund Reserve)」

  1. 回到剛才【檔案 1：結算傳票】已經篩選好的狀態，更改表頭 I/F Code 欄位的篩選，改為只包含 Refund Reserve（只選這個）。
  2. 將篩選結果貼到模板分頁 Refund Reserve。

Step 4. 排除 Refund Reserve 對當天總撥款的影響

  1. 分頁 Finance IF (Admin JE) 是當天整天要撥款的金額，因此必須將 Refund Reserve 剔除。
  2. 將 Refund Reserve 分頁中的「商家名稱」，逐筆去對應 Settlement Statistics 檔案 Sheet1 中的「商家名稱」。（因每小時統計，商家篩選後可能有多筆）。
  3. 核對總和條件：
    • 根據欄位名稱 Base Amt. for settle 核對，兩邊的總和金額必須一致。
    • 且 Scheduled settelement Date 必須與 Refund Reserve 分頁一致。
  4. 如果總和數字確認一致，則**刪除 Settlement Statistics 對應的 rows**（刪除包含了 Refund Reserve、不應存在於當天要撥款金額的列）。

Step 5. 建立 AP/AR 總表 (搬移欄位)

  1. 繼續使用【檔案 1：結算傳票】（清除所有篩選）。
  2. 將傳票檔案 Column A~M 每個 row 的數據貼到模板分頁 APAR 的 Column A~M 中。
  3. 將傳票檔案 Column N~AE 貼到模板分頁 APAR 的 Column O~AF 中。
    主要差異：模板 APAR 的 Column N 是另外建立的（不存在於傳票），所以從 Column O 開始貼，並將 Column O 表頭改名為 AP（應付）。
  4. 貼完後，透過 Column H 標頭 I/F Code 進行篩選，僅保留以下四個，其他的 rows 直接刪除：
    LPTW_EPI_Payment、LPTW_EPI_Payment_Aft、LPTW_EPI_Settle_Bal、LPTW_EPI_Settle_Cc
  5. 排序 Column AB 標頭 NEON Fees Field（由小到大）。
  6. 去篩選挑出「空格」的 rows（代表 AR 應收），有值的代表 AP（應付）。
  7. 將 Column O（AP）的值，搬移到 Column N（AR）。
⚠️ 待確認事項：確認模板 APAR 分頁中，這個「另外建立的 Column N」的正式表頭名稱是否就是 "AR"？

Step 6. 驗證 Method 樞紐

  1. 包含 method 1、method 2、method 3 三個分頁。
  2. 分別將該分頁中的樞紐統計表「重新整理」。
  3. 把數值接到右邊空白處去（三個分頁中已經有指定的貼上位置，最後一個 column 帶有比對公式）。
  4. 進行結果比對，公式結果如果是 0（無差異）代表正確。

Step 7. 匯入商家統計數據

  1. 打開【檔案 2：Merchant Statistics】，將 Sheet1 從 Column A~L 的所有 rows 複製，貼到模板分頁 Mercant Settle Statistics 的 Column A~L。
  2. 將 Sheet1 從 Column M~O 複製，貼到模板的 Column N~P。
  3. 因為模板的 Column M 增加了一個名為 verfiy 的欄位（自帶公式），貼上後必須將 Column M 每個 rows 都下拉套用到公式。

Step 8. 核對餘額與點數 (BALANCE & LINE_POINT)

  1. 打開模板分頁 statistics_pivot，做樞紐來源資料更新。
  2. 找出樞紐標頭「標籤列」中，值為 BALANCE 以及 LINE_POINT 的每個 rows。
  3. 將對應的 Column A~F 貼到模板 statistics_LP& External Account 分頁的 Column B~G。
  4. 在 Column A 填上對應的字串：
    • 對應 BALANCE 填入：LP Account-EPI Balance
    • 對應 LINE_POINT 填入：LP Account-LINE Points
  5. 到模板分頁 LP& External Account Verify 更新樞紐來源，結果貼到右方（已有指定位置），下拉最右邊 verify 公式檢查請款與撥款是否一致。
  6. 篩選出非 "0" 的情況（非 0 代表有問題，要找出原因）。

Step 9. 核對信用卡 (CREDIT_CARD)

  1. 回到分頁 statistics_pivot 更新樞紐。
  2. 找出標籤列 CREDIT_CARD 的明細，將對應的 Column A~F 貼到分頁 statistics credit Verify 的 Column A~F。
  3. 將已存在的 Column G verify 公式下拉做檢核，篩選出非 "0" 的情況並查明原因。

Step 10. PV All 與小數點四捨五入處理

  1. 到分頁 PV All 更新總共兩個樞紐統計表的來源資料。
  2. 將名為 Finace I/F 的樞紐結果複製，貼到分頁 Rounding off 的 Column Q 開始貼（覆蓋之前的表格）。
  3. 到 Rounding off 分頁的 Column Y，更新樞紐統計表 Refund reserve。
  4. 回到 PV All 分頁，將 Settle 樞紐貼到 Rounding off 的 Column A 覆蓋。
    (★ 去除小數點操作：將 Settle 樞紐資料複製 -> 貼到記事本 -> 從記事本貼回樞紐統計表覆蓋，實現四捨五入)
  5. 從 Settle 樞紐作篩選加總日期，將每個 Column B~E 透過公式 SUM 重算總和。
    • 例如：第一個 SUM row 7~95、第二個 SUM row 96~20076...以此類推。
  6. 將欄位 E5 公式改為 =B5-C5-D5；Column B5~D5 改為剛才套用 SUM 結果的加總（例如 =B6+B96+B20076，根據實際 row index 為主）。
  7. 截圖記錄每個日期總和對應的 row index 備用。
  8. 到模板的【待確認分頁】中的 Column H，依據每個撥款日期（參考 Col Q 標籤列上的日期），更新總共 5 個 table 的公式：
    • 第 1 個 table：row 7~9 公式分別更新為 B6/C6/D6 (舉例)
    • 第 2 個 table：row 17~19 公式分別更新為 B96/C96/D96 (舉例)
    • 依此類推更新完 5 個 table。
⚠️ 待確認事項：上述步驟「更新 當天的撥款報表 分頁 (未記錄) 的 column H」，該分頁的確切名稱是什麼？

Step 11. 產出報告截圖

  • 最後將統計出來的統計結果（有綠底色的）截圖，貼到模板分頁 Report 內，這也是最後要拿來做回報的結果。

Step 12. 異常排錯機制 (抓帳)

  • 如果有對應不起來的情況，去抓 raw data 來比對。
  • 路徑：Settlement > merchent settlement data > merchent settlement row data
  • 選擇結帳日當天 (02/27) 以及有差異的 MID (商家名稱) 去搜尋，下載明細查看 row data 結果是否與傳票相符，相符即為正確。
