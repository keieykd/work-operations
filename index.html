📝 LINE Pay 結算與撥款業務手冊

1. 會議摘要

核心聚焦於 LINE Pay 第三方支付與電子支付的金流底層邏輯。從消費者「授權(Auth)」、商家「請款(Capture)」，到系統生成「撥款底稿(Settle)」並打包「結算(Aggregation)」，最終產出傳票完成撥款。重點釐清了日結、週結、月結遇上「假日」的系統排程差異（日結順延，週/月結假日照跑），以及法規對於代收代付的限制，為後續複雜的財務對帳打下堅實的觀念基礎。

【第一部分：核心觀念與架構】

2. 完整工作流程 (Transaction Flow)

  1. 授權 (Auth)：消費者刷卡，收單行發出通知。此階段僅「圈存」額度，尚未扣款。
  2. 請款 (Capture)：交易確認成立，商家向系統請款。資料正式進入 LINE Pay 系統。
  3. 撥款底稿 (Settle)：Capture 發生後約 1 秒內，系統瞬間逐筆生成應撥款的「原始資料」。
  4. 結算 (Aggregation)：系統依據商家合約週期（日/週/月），將多筆 Settle 打包成「一整包」。
  5. 傳票 (Journal / JE)：結算完成後，系統產出對應的財務撥款傳票金額。
  6. 撥款給商家 (Settlement)：財務端核對傳票與實際銀行金流無誤後，將資金撥付至商家帳戶。(註：Payout 比較是商店跟用戶之間會用的英文，撥款給商家通用語為 Settlement)。

3. 金句與雷區 (Tips & Red Zones)

💡 金句：「我們部門要做的核心，就是確保 Capture 資料都有到 Settle，Settle 有結算出正確的傳票 (JE)，最後傳票金額跟實際銀行出去的金流要一模一樣。」

🚫 絕對不能犯的禁忌與雷區：
  1. 搞錯假日的「結算」邏輯：
    日結遇假日會順延；但是 「週結」跟「月結」不管是不是假日，系統都會硬生生結算下去！
    👉 長假防禦機制：過年或長假前，務必先把系統會在假日自動結算的電子傳票，預先送交財務系統排程，否則連假回來第一天會撥款大當機。
  2. 忘記「撥款日必定是工作日」：
    如果商戶是 W111（週一結算，週二撥款），但週二是國定假日，這筆錢絕對不會在週二出去，系統會自動順延到週三（下一個工作日）撥款。
  3. 搞混「代收代付」的法規限制：
    第三方支付不能做代收代付業者跟代收代付業者的介接。但是電子支付可以做，法規有開一個後門，給「外送平台」還有「計程車」，可以代收代付業者跟代收代付業者接，僅限這兩個。
    因此，我們不能幫純外送平台（如 Uber、FoodPanda）做全額代收代付。對 Uber 我們僅做「資訊串接與分潤抽成」，不碰本金；但如果是 Momo（買貨進來再賣出去的實質買賣），我們就可以完整代收代付。

5. 專有名詞小字典

專有名詞白話解釋
第三/電子支付差異 第三方支付 (TSP)：只有信用卡支付。需串接收單行(玉山/台新等五大行)，錢從銀行來。
電子支付：區分成「用信用卡支付」或「用儲值金支付」。如果用信用卡支付就有收單；用儲值金就沒有收單，直接扣款。
★ 系統判定邏輯：消費時，若商家與用戶雙方皆有申請電子支付，系統即走電子支付；若有一端未申請，就會走 TSP (第三方支付)。
Auto / Manual CaptureAuto：授權後立刻請款。Manual：非即時請款。例如：7-11 交易量太大一天包成一包請款；NET 每 4 小時請款一次；Uber 等餐點送達後才請款。
MID (Merchant ID)商店代號。同集團（如新光三越）統編相同，但為區分不同電腦或櫃位，會擁有多個不同 MID，通常前綴號碼會一樣。
日結 (A1, A2...)A1 = 次 1 日撥款；A2 = 次 2 日撥款。★ 日結僅在工作日進行結算。
週結 (W110, W111...)W 代表週結，統一結算上週一到週日的帳。W110 = 週一當天撥款；W111 = 週二撥款。
月結1 號到月底的帳，統一在次月 1 號結算（次月 2 號或 5 號撥款）。

【第二部分：每日結算統計報表 SOP (實戰操作)】

前情提要：假設今天為 02/27，要做今天的結算統計報表。操作分為「下載原始檔」、「清除舊資料」與「整合至當天撥款報表模板」三大階段。

一、 下載三份來源檔案 (Raw Data)

📥 檔案下載與統整原則：
• 統整用的總檔案，通常會用上週一的作為參考來源，因為資料比較完整（會包含週結資料進來），例如：LINE Pay EPI_Settlement Report_20260331。
• 從系統下載後的檔名日期可能變動，但開頭英文字不變。
• Bulk Download 的共通路徑為： Admin Function > Bulk Data Download 尋找生成的檔案。
1. 下載傳票 (Search for Merchant Settlement)
  • 下載檔名開頭： MERCHANT_SETL_AGRN_BASE_...
  • 路徑：Admin Function > Settlement Confirm > Search for Merchant Settlement
  • 搜尋條件：
    • Aggregation date: 選擇當天日期 02/27
    • Scheduled Settlement date: 選擇範圍為 當天 02/27 - 往後兩個月 (例如 04/30) (主要是要把最久的 A25 情況包含進來)
  • 操作：按下 Search 生成結果後，點選 Bulk Download，並勾選 Delete previous aggregated file，再點選生成按鈕，以下載結算傳票 Excel。
2. 商家統計數據 (Merchant Statistics)
  • 用途：抓取每個商家的統計數據，比對請款與撥款是否一致。
  • 下載檔名開頭： MERCHANT_SETL_STATS_...
  • 路徑：Settlement > Settlement Statistics > Merchant Statistics
  • 搜尋條件 (交易日)：搜尋前一天 (例如 02/26-02/26)。
    ★ 特殊規則：如果前一天是假日，則要包含前一個最後一個工作日，以及中間包含的所有假日 (例：今天是週一，則選擇上週五~週日的日期範圍)。
  • 操作：點擊黃色的 Bulk download 鈕。在跳出的 dialog 選擇日期範圍，必須勾選 Delete previous Aggregation file。非同步下載，須至 Bulk Data download 等待寫入完成下載。
3. 逐筆撥款明細統計 (Statistics for validation)
  • 下載檔名開頭： MERCHANT_SETL_VALID_AGRN_...
  • 路徑：Settlement > Settlement Statistics > Statistics for validation
  • 搜尋條件 (日期範圍)：選擇結算日當天 02/27。
  • 操作：同樣點擊黃色的 Bulk download 鈕，選擇日期範圍並勾選 Delete previous Aggregation file。至 Bulk Data download 清單等待下載完成。

二、 清除舊資料前置作業

先把「當天的撥款報表」(例如：LINE Pay EPI_Settlement Report_20260331) 這個檔案進行部分清除動作，主要是清除原有的來源資料，不動樞紐的分頁以及公式欄位。刪除的分頁以及內容如下：

  • Mercant Settle Statistics：刪除 Column A~O。
  • APAR：刪除 Column A~AG。
  • Refund Reserve：刪除 Column A~AF。
  • Finance IF (Admin JE)：刪除 Column A~AF。
  • Settlement Statistics：刪除 Column A~M。
  • Report：刪除圖片 (從 Rounding off 截圖過來的樞紐統計表)。
  • statistics_LP& External Account：刪除 Column A~G，從 row 3 開始刪除 (row 2 是表頭不能刪)。

三、 整合至「當天的撥款報表」Excel

以下操作皆在剛清除好舊資料的「當天的撥款報表」模板中執行。

Step 1. 處理「逐筆撥款明細」 (排除後結手續費)

  1. 打開檔案 3 MERCHANT_SETL_VALID_AGRN，把 Sheet1 全部的值貼到模板的分頁 Settlement Statistics 中（連表頭都一樣不變）。
  2. 將 Column Fee Charge Method 篩選出值為 Payable。
  3. 接著把 Service Fee 跟 VAT 這兩個 Column 篩選出來的每個 rows 的金額都刪除（因為這算是手續費後結，不放進現在的資料表內）。

Step 2. 處理「結算傳票」(篩選 AP 應付與整理格式)

  1. 打開檔案 1 MERCHANT_SETL_AGRN_BASE 進行資料篩選：
    • 篩選表頭 NEON Fees Field 欄位：篩選出包含 Fee 字串（代表 AP-應付），不勾選空白的選項。
    • 篩選表頭 Transferred(Y/N) 欄位：選擇 Y（代表已傳送至 NEON 系統，應付為正數）。
    • 篩選表頭 I/F Code 欄位：僅挑出 LPTW_EPI_Payment 及 LPTW_EPI_Payment_Aft，只選這兩個，其他不選。
  2. 將這個表格的全部篩選結果，貼到模板的分頁 Finance IF (Admin JE)。
  3. 貼上後，再次在該分頁篩選表頭 I/F code 為 LPTW_EPI_Payment_Aft。接著把標頭為 Service Fee & VAT(Service Fee) (對應 Column O & P) 這兩個 Column 的每個 rows 金額都刪除（同樣為手續費後結）。

Step 3. 處理「退款準備金 (Refund Reserve)」

  1. 回到剛才已經篩選好的 MERCHANT_SETL_AGRN_BASE 檔案，更改表頭 I/F Code 欄位的篩選，改為只包含 Refund Reserve（只選這個）。
  2. 將篩選結果貼到模板分頁 Refund Reserve。

Step 4. 建立 AP/AR 總表 (搬移欄位)

  1. 繼續使用 MERCHANT_SETL_AGRN_BASE（清除所有篩選）。
  2. 將傳票檔案 Column A~M 每個 row 的數據貼到模板分頁 APAR 的 Column A~M 中。
  3. 將傳票檔案 Column N~AF 貼到模板分頁 APAR 的 Column O~AG 中。
    主要差異：模板 APAR 的 Column N (AR) 是另外建立的（不存在於傳票），所以從 Column O 開始貼，並將 Column O 表頭改名為 AP（應付）。
  4. 貼完後，透過 Column G 標頭 I/F Code 進行篩選，僅保留以下四個，其他的 rows 直接刪除：
    LPTW_EPI_Payment、LPTW_EPI_Payment_Aft、LPTW_EPI_Settle_Bal、LPTW_EPI_Settle_Cc
  5. 排序 Column AB 標頭 NEON Fees Field（由小到大）。
  6. 去篩選 Column AB 標頭 NEON Fees Field，挑出「空格」的 rows（代表 AR 應收），有值的代表 AP（應付）。
  7. 將 Column O（AP）的值，搬移到 Column N（AR）（也可將 AP 先複製到 AR 後，再刪除 AP 原有資料）。

Step 5. 驗證 Method 樞紐

  1. 包含 method 1、method 2、method 3 三個分頁。分別將該分頁中的樞紐統計表「重新整理」。
  2. 把數值接到右邊空白處去（三個分頁中已經有指定的貼上位置，最後一個 column 帶有比對公式）。
  3. 進行結果比對，公式結果如果是 0（無差異）代表正確。
    補充說明：
    分頁 method 2 與 method 3 中，LPTW_EPI_Payment 及 LPTW_EPI_Payment_Aft 代表 AP (同一個 row 只會有一個 cell 有值)；LPTW_EPI_Settle_Bal 及 LPTW_EPI_Settle_Cc 代表 AR。每個 row 的 AP 總和必須等於 AR 總和。
    公式會用其中一個 column 作為代表，可能要手動修改公式參照（例如：該 row 預設公式是 =H5-J5-K5，但如果 AP 的值出現在 Column I，則需手動改為 =I5-J5-K5）。

Step 6. 匯入商家統計數據

  1. 打開檔案 2 MERCHANT_SETL_STATS，將 Sheet1 從 Column A~O 的所有 rows 複製，直接貼到模板分頁 Mercant Settle Statistics 的 Column A~O。

Step 7. 核對餘額與點數 (BALANCE & LINE_POINT)

  1. 打開模板分頁 statistics_pivot，做樞紐來源資料更新。
  2. 找出樞紐標頭「標籤列」中，值為 BALANCE 以及 LINE_POINT 的每個 rows。
  3. 將對應的 Column A~F 貼到模板 statistics_LP& External Account 分頁的 Column B~G。
  4. 在 Column A 填上對應的字串：
    • 對應 BALANCE 填入：LP Account-EPI Balance
    • 對應 LINE_POINT 填入：LP Account-LINE Points
  5. 到模板分頁 LP& External Account Verify 更新樞紐來源，結果貼到右方（已有指定位置），下拉最右邊 verify 公式檢查請款與撥款是否一致。
  6. 篩選出非 "0" 的情況查明原因。
    補充說明：標籤列如果是 AG 開頭的代表是代理商群組名稱，不會有 Captured 的值，所以 Verify 計算出來不是 0 的情況是正常的。

Step 8. 核對信用卡 (CREDIT_CARD)

  1. 回到分頁 statistics_pivot 更新樞紐。
  2. 找出標籤列 CREDIT_CARD 的明細，將對應的 Column A~F 貼到分頁 statistics credit Verify 的 Column A~F。
  3. 將已存在的 Column G verify 公式下拉做檢核，篩選出非 "0" 的情況並查明原因。
🔍 異常查明方式 (以 PChomeEC 為例)：

1. 進入路徑：Settlement > Merchant Settlement data > Merchant settlement raw data。
2. Period 選擇 Scheduled Aggregates Date 並選擇結算當天日期，Merchant ID 填上 PChomeEC，Search 後 Download。
3. 打開 Excel，將 Payment method 篩選取出 CREDIT_CARD 的 rows。
4. 確認 Column Base Amt. for settle 總和是否與原本檔案中 Column 加總 - Base Amt. for settle 相符。若相符則無異常（但需向韓國方回報此問題）。

Step 9. PV All 與小數點四捨五入處理

  1. 到分頁 PV All 更新總共兩個樞紐統計表的來源資料。
  2. 將名為 Finace I/F 的樞紐結果複製（在右方的樞紐，將樞紐統計表根據標籤列中的日期收合，再貼過去），貼到分頁 Rounding off 的 Column Q 開始貼（覆蓋之前的表格）。
  3. 到 Rounding off 分頁的 Column Y，更新樞紐統計表 Refund reserve。
  4. 回到 PV All 分頁，將 Settle 樞紐貼到 Rounding off 的 Column A 覆蓋。（從 title 標籤列下方的日期開始貼，展開不要收合）。
    ★ 去除小數點與凍結數值操作（取代記事本大法）：
    為了後續能根據斷行精準加總，必須將樞紐分析表轉換為純數值：
    1. 框選 Settle 樞紐資料並按 Ctrl+C 複製。
    2. 原地按右鍵，選擇「貼上值 (Paste as Values)」。
    3. 選取這些貼上的數值，按 Ctrl+1 進入儲存格格式，將「數值」的小數點位數設為 0 實現四捨五入。
    ⚠️ 警告：此動作會破壞該處的樞紐分析表結構（將其變成死數字），這是為了後續手動 SUM 加總的必要防呆操作。
  5. 接著將標籤列進行篩選出日期以及總計 (勾選新增目前的範圍進篩選)，並將 column B rows 重新進行加總，以實際 index 範圍為主，接著公式套用到 column C & column D。
    • 例如：第一個 SUM row 7~95、第二個 SUM row 96~20076...以此類推。
  6. 最後再更新上總計：將欄位 E5 公式改為 =B5-C5-D5；Column B5~D5 改為剛才套用 SUM 結果的加總（例如 =B6+B96+B20076，根據實際 row index 為主）。
  7. 截圖記錄每個日期總和對應的 row index 備用。
  8. 到模板的 Rounding off 分頁中的 Column H，依據每個撥款日期（參考 Col Q 標籤列上的日期），更新總共結算週期對應的 5 個 table 公式：
    • 第 1 個 table：row 7~9 公式分別更新為 B6/C6/D6 (舉例)
    • 第 2 個 table：row 17~19 公式分別更新為 B96/C96/D96 (舉例)
    • 依此類推更新完 5 個 table。
💡 補充更新項目：
1. 更新表格日期：最後也要更新上這 5 個 table 的日期，日期需對應到結算週期 (A1/A2/A3/A7/A25) 這 5 個 table 的新日期，分別填寫在每個 table 的左上角。
2. 更新今日日期：欄位 G4 也要更新為今天日期。
3. 更新 Refund：另外也要逐個更新上這 5 個 table 的 column I (Refund)，根據 column Y 的樞紐統計表，對應到結算週期 (A1/A2/A3/A7/A25) 這 5 個 table。

Step 10. 產出報告截圖

  • 最後將統計出來的統計結果（位於 Rounding off 分頁中 A1/A2/A3/A7/A25 這 5 個 table 內的綠底色儲存格）截圖，貼到模板分頁 Report 內，這也是最後要拿來做回報的結果。記得更新回報日期。

Step 11. 異常排錯機制 (抓帳)

  • 如果有對應不起來的情況，去抓 raw data 來比對。
  • 路徑：Settlement > merchent settlement data > merchent settlement row data
  • 選擇結帳日當天 (02/27) 以及有差異的 MID (商家名稱) 去搜尋，下載明細查看 row data 結果是否與傳票相符，相符即為正確。

Step 12. 發送 EMAIL 報告

  • 將剛才 Report 分頁的報表截圖貼上 Email，注意：貼上後要把圖拉大，確保數據清晰可見。
  • Email 格式資訊：
    主旨範例：[LPEPI] [Billing] Report of merchant settlement_2026.04.07 收件者：TW Pay EPI Finance <dl_twpayepifin@linecorp.com> 副本：EPI Billing <dl_twepibilling@linecorp.com>

【第三部分：實際撥款與傳票操作流程】

1. 結算核准與傳票發送 (Approval & Journal)

  1. 進入路徑：Administrator Function > Settlement Confirm > Approval of merchant settlement。
  2. 設定搜尋條件：
    • Scheduled Settlement Date：選擇結算日到往後兩個月 (例如 03/31 到 05/31)
    • Aggregation Date：選擇結算當天 (例如 03/31)
  3. 點選 Search 後，點選 Bulk Download 按鈕（勾選 Delete previous aggregated file）。
  4. 至 Bulk Data Download 下載 Excel，檔名為 MERCHANT_SETL_AGRN_BASE_... 開頭。
  5. 打開檔案進行篩選核對：
    ① NEON Fees Field 非空白 (勾選有值)
    ② Transferred(Y/N) 選擇 Y
    ③ I/F Code 選擇 LPTW_EPI_Payment 以及 LPTW_EPI_Payment_Aft
  6. 加總 Actual Deposit Amount，並核對總和值是否等同於統計結果（Rounding off 分頁中 A1/A2/A3/A7/A25 內的綠底色儲存格）。
    ※ 確認完後，這份 MERCHANT_SETL_AGRN_BASE 檔案純屬檢核用，後續不再使用。
  7. 確認無誤後，回到 Approval of merchant settlement 的 Approve merchant settlement 標籤。選擇 Aggregation Date 為今天，Scheduled Settlement Date 選擇兩個月，點選 SEARCH，接著按下紅色邊框的 OK。
  8. 發送 LINE 訊息：在群組 "EPI Accounting / Billing" 通知：
    @Norah Lien @雅菁 Eileen 早安, 撥款報表以及JE都已經寄出囉, 多謝
  9. 等待 15~30分鐘，於 Approval of merchant settlement 介面再次 Search，確認下方傳票已傳送（剩下 AG 開頭的 MID，或觀察分頁數減少）。下載檔案確認 Transferred (Y/N) 欄位皆為 N 即代表傳送完畢。

⏳ ========= 等待時間要做的事情 (重送 Retry 作業) =========

Step A: 填寫 Transfer Summary List 檔案
填寫檔案 Transfer Summary List_yyyymm，將結算週期代碼 (A1~A25) 的值填上對應日期。

Step B: 處理重送 (Re-send) 金額與日期
當天清單最後需加上 Re-send。根據郵件 Merchant Modification Request (MM檔案) 內容，撥款失敗原因分兩種：
1. 商家資料不齊全 (銀行帳號不存在等)：等待更新後再重新撥款。
2. 退款水位不足：此情形每天都要重送（即重送上一個工作日失敗的紀錄）。

Step C: 篩選 MM 附件並填寫重送日
打開附件篩選以下兩分頁：
• Failed Transfer_MM 分頁：篩選 Settlement Complete Date 為「空白」，且 Status 包含 retry 或 update completed。
• Insufficient Money Pool 分頁 (退款水位不足)：篩選 Settlement Complete Date 為「空白」或已填上「當天日期」。
👉 動作：將上述需重送的 rows，在 Settlement Complete Date 欄位填上「今天要重送的日期」。

Step D: 統計重送總額
將 Settlement Complete Date 篩選為重送當天日期，加總兩個分頁的 Actual Transfer Amount。加總後填寫至 Transfer Summary List_yyyymm 的 Re-send 總和中。

Step E: 系統上執行 Retry
至 Administrator Function > Settlement Confirm > Transfer approval 下拉到 Transfer Request List 執行重送：
▶ 狀況1 (退款水位不足)： Scheduled Settlement Date 一般選擇「前一天撥款日」(依據 MM 檔案為準)，Status 選擇 Insufficient Pool。Search 後，絕對不能點選 Retry(All)。必須在每一頁勾選清單，然後點選 Retry(selected)，逐頁勾選送出。（否則系統會把當天的全送，包含不是「退款水位不足」的情況都會一起送出造成錯誤）。
▶ 狀況2 (商家資料不齊全)： 依據 MM 檔案日期搜尋 Scheduled Settlement Date，加上單筆 MID 條件，Status 選擇 Failed。於清單中點擊該項目最右方的 Retry。

2. 異常排除：傳票傳輸失敗

  1. 如何檢查有無異常：到 Approval of merchant settlement 中 Scheduled Settlement Date 選擇結算日當天，並下載 Download 結果。篩選 Transferred(Y/N) 的結果，如皆為 N 就是完成了。
  2. 若傳票(JE)有傳輸失敗（例如沒有變成 N），至 Settlement > Finance I/F 搜尋結算當天 Period 及 Sent Status 為 Fail 的紀錄查看原因。
  3. 常見問題：缺少 vendor code / vendor ID
    • 用統編向財務申請。至 Merchant Management > Basic information > Merchant inquiry 透過 MID 查統編。
    • 接著至 Settlement > Merchant Settlement data > Merchant Vendor ID I/F，用 MID 搜尋並 Edit 設定 vendor code/vendor ID (預設 Success)。
  4. 針對錯誤進行重送：回到 Approval of merchant settlement 的 Approve merchant settlement。Aggregation Date 選今天，Scheduled Settlement Date 選兩個月，輸入 MID 點擊 SEARCH，按下紅色 OK。

3. 銀行屬性設定與撥款清單下載

傳票送交財務後 等待 15 分鐘 再進行此步驟（即按下 Approve merchant settlement 的 OK 鍵 15 分鐘後）。

  1. 至 Administrator Function > Settlement Confirm > Transfer approval 下拉至 Transfer Request List。
  2. Scheduled Settlement Date 選擇撥款日(當日)，點選 Bulk Download（勾選 Delete previous aggregated file）。
  3. 下載 MERCHANT_SETL_PAY_REQ_SCHDL_... 檔案。檢查 Actual Transfer Amount 總和是否等於今日應撥款總和 (核對 Transfer Summary List_yyyymm)。
    👉 若不一致：稍等片刻再重新下載。
  4. 若一致，進行非合作銀行 (FISC-Exception) 批量設定：
    篩選 Bank Type 為 COOPERATIVE_BANK，檢查銀行名稱是否包含「台新國際商業銀行」或「合作金庫商業銀行」或「高雄銀行」。若有，將 MID 複製至 txt 檔。
    前往 Operation Menu > Settlement Master Data > Settlement Banking Setup，點選 Register(Bulk)，Settlement Banking System 選擇 FISC-Exception 並上傳 txt 檔。
  5. 進行非合作銀行大額 (CTBC+CRPD) 個別設定：
    篩選 Bank Type 為 NON_COOPERATIVE_BANK，檢查是否有單筆 Actual Transfer Amount > 200 萬。若有，挑出 MID 進行個別設定：
    點擊 Register(Each) 填上 MID，選擇 CTBC+CRPD。(可先進行 MID 搜尋確認清單狀況)。
    ※ 目的：提前修改銀行屬性，避免撥款失敗。

4. 撥款放行與回報財務

  1. 至 Transfer approval > Transfer Summary List 選擇 Scheduled Settlement Date 為撥款日(今日)，搜尋並下載。
  2. 確認 Total Transfer Amount 總和與今日撥款總和相符。
  3. 將 Column B~E 的值貼至 Transfer Summary List_yyyymm 右半部。
  4. 截圖並在群組 (EPI Accounting / Billing) 通知 @Aimee Lee，內文：
    再麻煩協助 verify payment, 謝謝。
  5. 等 Aimee 回覆 OK 後，回到 Transfer Summary List 搜尋撥款當天，點選 Approve All 按鈕 (需等待系統處理)。
  6. 約 1 小時後，回系統查找是否有失敗狀況。
⚠️ 系統撥款三大途徑與核心觀念：
1. 合作銀行 (中信、聯邦等)。合作銀行清單參考：Wiki 連結
2. 非合作銀行：走財經 (FISC)。
3. 管理銀行 (CTBC-CRPD)：當非合作銀行的單筆金額「大於 200 萬」時，強制走此通道。

🚨 注意：若有透過 CRPD 重送的項目，務必在下午 3:30 前完成處理！

5. 失敗情況處理與系統錯誤重送 (System Error)

撥款失敗通常分為三種情況：

  • 情況 1：退款水位不足 ➔ 將資料填上 MM 檔案。
  • 情況 2：商家資料不齊全 (帳號不存在等) ➔ 將資料填上 MM 檔案。
  • 情況 3：系統錯誤 ➔ 當天必須再次重傳 (見下方詳細流程)。
🔧 情況 3 (系統錯誤) 重送詳細操作流程：

第一步：確認系統狀態與實際金流

  1. 至 Transfer Request List 搜尋撥款日(今日)，查看 Status 為 Verified / Approved / Processing 是否有未處理完的結果。
  2. 若是 Verified / Approved 有項目：點選讓系統去處理。
  3. 若是 Processing 有項目 (需判斷是否已撥款成功)：
    • 若是彰化銀行 (CHB) 且為合作銀行：登入網銀查詢 (活期性存款明細查詢/下載檔案 CSV)，核對「已成功撥款金額(*0.01)」及「萬用註記」與系統後四碼是否相符。
    • 若是中信 (CTBC)：至 Slack 詢問 CJ 或 Nicole。
    • 若是中信合作銀行卡在 Processing：只能等下一個工作日中信提供對帳單（可請 Aimee 幫忙查詢）。若確認未匯出，狀態改失敗並安排重送；若成功則狀態改為成功。
    • 若確認沒有匯款成功紀錄：在系統上更改狀態為 To-be: Failed。

第二步：分析 Failed 原因與銀行轉換

  1. 至 Transfer Request List 搜尋當日 Failed 狀態，下載 EXCEL。
  2. 篩選 非合作銀行 (Bank Type)，逐一用 MID 了解原因 (是否之前就 Failed 或今日才發生)。
  3. 官股銀行 + 專戶 (如台灣銀行、學校專戶)：
    至 Settlement Banking Setup 搜尋該 MID 看是否需 CLOSE，將其重新設定為 CTBC+CRPD。回到 Transfer Request List 點選 Retry。
    通知 @Aimee 核准 ("Hi Aimee 剛才有一筆重送 再麻煩核准, 多謝。")。核准後過濾 Verified 狀態，點選 approve 送出。
    💡 特別注意：若透過 CTBC-CRPD 成功撥款，需發信給 CTBC 確認是否有退匯紀錄 (Subject: [LINE Pay EPI] 退匯交易協助確認_0401)。
  4. 當合作銀行出現的錯誤情況與處置：
    • [DATA_VALIDATION_ERROR]：將撥款方式改為財經。
    • [FAILURE]：不需改設定，稍等再次重送。
    • [EXTERNAL_SYSTEM_ERROR]：先至網銀登入再次查詢驗證，避免重複撥款。確定無撥款紀錄則不需改設定，稍等再次重送。
    • [RESTRICT_BANK_ACCOUNT_ERROR]：將撥款方式改為財經 FISC。
  5. 王道銀行 (O-Bank)：
    無法撥款給個人商戶(無統編)。若 MID 為個人商戶，直接改為財經 FISC。

第三步：執行重送與二次核准

  1. 確認完畢後，開始進行合作銀行重送。逐筆依 MID 點選 Retry (每次重送需核對金額)。
  2. 全部 Retry 完成後通知 Aimee 核准 ("Hi Aimee 剛才有10筆重送 再麻煩核准, 多謝。")。
  3. Aimee 核准後，過濾出 Verified 狀態，執行 Approval all。
  4. 再次檢查是否皆成功。若仍有 FAILED 且屬系統問題，再送當天最後一次 (即第三次重傳)。

6. 日終歸檔與 MM 報表發送

  1. 查詢並下載當天最後的 FAILED Excel 檔案。
  2. 下載 Insufficient Pool 的 Excel 檔案。
  3. 下載 Paid 的 Excel 檔案 (資料量較多)。
  4. 更新 Transfer Summary List 檔案 (優先放入失敗情況)：
    打開 Transfer Summary List_yyyymm，將成功 (Paid) 以及上述兩種失敗的情況紀錄上去：
    • 成功的貼至專屬分頁 Transfer Success
    • 失敗的貼至專屬分頁 Failed Transfer_MM
    • 成功+失敗的合併貼至分頁 Transfer Request List(Suc+Fail)
  5. 製作 MM (Merchant Modification Request_EPI) 檔案：
    將剛才整理好的「退款水位不足」與「商家資料不齊全」等失敗資料，分別填入 MM 檔案對應欄位，並將下一個工作日 (要 Retry 的日期) 先打上。
  6. 發送退匯交易確認信與 MM 報表：
    發送信件給 CTBC 確認退匯交易。等待中信回覆確認沒有退匯紀錄後，即可將 MM 檔案發信送出。
    ✉️ 郵件主旨範例：[LPEPI] [Billing] Merchant Modification Request_20260402

7. 沖帳傳票發送 (Clearing JE Confirm)

  1. 在最後一步驟，確認當日撥款皆已完成，且確認沒有退匯交易後，即可執行此動作送出沖帳傳票給財務。
  2. 至 Transfer approval > Transfer Request List 介面。
  3. 將 Scheduled Settlement Date 篩選為撥款日當日。
  4. 將 Status 篩選為 Paid，然後點選 Search。
  5. 確認清單無誤後，點選頁面上的 Clearing JE confirm 按鈕送出。
